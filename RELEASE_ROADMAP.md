## 1. 全体像

```
Phase A: 公開前必須固め（法務・データ主体の権利・インフラ確定）
    ↓
Phase B: 品質保証体制（自動テスト・CI）
    ↓
Phase C: 運用・監視体制（本番公開はここで解禁）
    ↓
Phase D: リリース後グロース（任意・優先度は公開後に再検討）
```

A→B→C は原則直列（B・Cの土台がないまま一般公開するとインシデント対応ができない）。
Phase D は公開後の学びを踏まえて再設計するため、今は方向性のみ示す。

## 2. Phase A: 公開前必須固め

「一般ユーザーに使ってもらう」ために避けて通れない項目。ここが終わるまでは
限定公開（自分・知人のみ）にとどめることを推奨。

### REL-01: F-1（完了・2026-07-18）

**決定・実装済み**: 誕生日の自己変更は2回まで、3回目以降は管理者への
問い合わせが必要（`UserProfile.birthDateEdits` ＋ rules の
`birthDateEditsOk()`。既存の誕生日を消してからの再設定で回数をリセット
することも禁止）。あわせて誕生日入力を `<input type="date">` から
年/月/日の3セレクト（[BirthDatePicker](mindmap-app/src/components/BirthDatePicker.tsx)）
に置き換え、古いカレンダーを何十年分もめくる手間を解消した。

### REL-02: アカウント削除・データ削除機能（法的に実質必須）

個人情報保護法上、利用者からの開示・訂正・削除請求に応じる仕組みが必要。
現状は退会導線が存在しない。

- 設定画面に「アカウントを削除する」を追加
- 削除対象: Firebase Auth アカウント、users/{uid} プロフィール、
  bookmarks サブコレクション、自分が所有する maps、自分の posts/comments
  （投稿は「削除済みユーザーの投稿」として匿名化する案 or 完全削除する案を選ぶ）
- 実装は Cloud Functions の onCall（本人確認込み）で一括処理

### REL-03: 利用規約・プライバシーポリシー

- 収集する個人情報（email・表示名・誕生日/年齢・投稿内容）と利用目的を明記
- AI（Groq）へ送信するデータの範囲を明記（テーマ・ノード内容・年齢帯）
- 未成年ユーザー（5歳〜）が想定利用者に含まれる旨と、保護者向けの説明
- お問い合わせ・削除請求の窓口（REL-04 と連携）
- **法律文書のため、雛形を用意した上で最終確認は専門家に依頼することを推奨**

### REL-04: サポート・問い合わせ導線

- フッター等に問い合わせ先（メールアドレス or フォーム）を設置
- 不具合報告・削除請求の受け口として最低限必要

### REL-05: 本番インフラ確定

- **Vercel**: プロジェクト作成、カスタムドメイン設定、環境変数登録
  （`GROQ_API_KEY`＝匿名フォールバック経路用、`NEXT_PUBLIC_FIREBASE_*`、
  `NEXT_PUBLIC_AI_BACKEND=functions`）
- **Firebase**: Authentication の「承認済みドメイン」に Vercel の本番ドメインを追加
  （これを忘れると本番でログインが失敗する）
- **Firebase デプロイ**: `firebase deploy --only firestore:rules,functions`
  （SEC-01 のルール修正・NF-05 のプロンプト変更が未反映のまま）
- dev/prod 分離は当面見送り、**デプロイ前チェックリストで代替**
  （無料運用中は Firebase プロジェクトを1つに保ち、変更前に rules を
  ローカルでレビューする運用でリスクを抑える。利用者が増えたら分離を再検討）

### REL-06: 本番の安全対策の有効化

- **App Check**（reCAPTCHA v3）を Functions（`enforceAppCheck: true`）と
  クライアント両方に導入。無料枠の Groq/Firebase を Bot 消費から守る
- Groq・Firebase の**予算アラート**を設定（使用量が想定を超えたら通知が来る状態にする）
- 現行のレートリミット（Functions 30回/時・API Routes IP単位30回/時）は
  本番トラフィックで再検証

## 3. Phase B: 品質保証体制

Phase A 完了後、一般公開の前に「壊れていないことを機械的に確認できる」状態を作る。

### REL-07: 最小限の自動テスト

全機能を今すぐ網羅する必要はない。まず壊れたら困る核だけ:
- `gauge.ts`・`ai-validate.ts`・`merge.ts`（純関数、ロジックの心臓部）の単体テスト
- Firestore rules のユニットテスト（`@firebase/rules-unit-testing`。
  SEC-01 で見つけたような穴を今後は自動で検出できるようにする）
- 主要フロー（マップ作成→ノード追加→AI提案→レビュー）の E2E テスト1本
  （Playwright 推奨）

### REL-08: CI 導入

- GitHub Actions で PR ごとに `lint` → `tsc --noEmit` → `build` → テスト を自動実行
- Firestore rules は CI 上でエミュレータを立てて検証（ローカルに Java が
  無く手動検証できていない問題も解消する）

## 4. Phase C: 運用・監視体制

**ここまで終えたら一般公開の判断ができる状態。**

### REL-09: エラートラッキング

- クライアント・API Routes・Functions の例外を収集する仕組み（Sentry 等）を導入
- 現状は `console.error` のみで、本番の不具合に気づく手段がない

### REL-10: 利用状況モニタリング

- Firebase Analytics か GA4 を導入し、主要フロー（マップ作成数・AI利用数・
  完成率・コミュニティ投稿数）を追える状態にする
- コスト系（Groq 消費量、Firestore 読み書き回数）を定期確認する運用を決める

### REL-11: 障害対応の型化

- デプロイ前チェックリスト（REL-05 と統合）
- ロールバック手順（Vercel は前デプロイへのロールバックが容易、
  Firestore rules も直前バージョンへ戻せることを確認しておく）
- 依存パッケージ監査（SEC-07 の `npm audit`）を月次など定期実行に
