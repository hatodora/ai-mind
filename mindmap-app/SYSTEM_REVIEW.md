## 残存リスク（Phase 4 以降の検討事項）

1. **ゲージ・バッジ・完成フラグはクライアント権威**
   Firestore ルールは型しか見ないため、悪意あるユーザーは自分のデータを直接書き換えて
   ゲージ満タン・全バッジ獲得にできる。個人の記録なので実害は小さいが、
   NF-01（コミュニティ）でバッジ等を「他人に見せる」なら Functions 側での再計算・検証が必要。

2. **匿名AI経路（API Routes）にレートリミットが無い**
   入力検証は入れたが、正しい形のリクエストを大量に送られると Groq 無料枠を消費される。
   Netlify のサーバーレスではメモリ内リミッターが効かないため、本質対策は
   (a) 本番を Functions 経路に一本化（NEXT_PUBLIC_AI_BACKEND=functions 設定済みなら
   ログインユーザーは移行済み。匿名向け Routes を残すかは製品判断）、
   (b) Netlify のレートリミット/Edge 機能の利用。

3. **プロンプトインジェクション**
   ノードラベル・テーマはそのままプロンプトに埋め込まれる。出力はテキスト表示のみで
   XSS 等には繋がらないが、変な提案を返させることは可能。実害は低い。

4. **同時編集は最後の保存が勝つ（last-write-wins）** → **Phase 4 で対応済み**
   onSnapshot 購読＋ノード単位マージ（src/lib/merge.ts、ダーティ窓5秒）を導入。
   完全な CRDT ではないため、極端な同時編集では稀に片方の変更が落ちる可能性は残る。

5. **aiCache が無限に貯まる**
   期限切れ後もドキュメントは残る。Firestore の TTL ポリシー（expiresAt を timestamp 型に
   変えて設定）か、定期削除 Function を Phase 4 で追加推奨。

6. **プロフィール読み込み失敗が「未登録」と区別されない**
   一時的な通信エラーでも /setup に誘導され、再登録すると createdAt 等がリセットされる。
   頻度は低いが、エラー時は再試行UIを出す方が安全。

7. **firestore.rules: 管理者が自分を role='user' に戻せてしまう**（roleUnchanged の仕様）。
   実害なし・仕様として許容。

8. **ホーム画面のマップ削除ボタンが `<Link>` の内側**
   動作はするが HTML として不正・スクリーンリーダーに不親切。UI改修時に外へ出す。
